# 429 限速 & 子代理失败 根因分析与解决方案

> 调研日期：2026-03-21
> 调研人：太宗

---

## 一、问题全景

### 1.1 429 Rate Limit（API 限速）

**现象**：`⚠️ API rate limit reached. Please try again later.`

**今天发生次数**：至少5次以上

**触发场景**：
| 场景 | 原因 | 发生次数 |
|------|------|---------|
| 同时 spawn 多个子代理 | 多个 session 同时向同一 API 发请求 | 3次以上 |
| cron 间隔太短（4分钟） | 上一个还没结束，下一个已触发 | 2次 |
| 主 session + 子代理同时活跃 | 两个 session 并行消费 API 配额 | 间歇性 |

### 1.2 Subagent Failed（子代理失败）

**现象**：`❌ Subagent gm failed`

**今天发生次数**：至少4次

**触发场景**：
| 场景 | 实际错误 | 发生次数 |
|------|---------|---------|
| 任务太大（读多个大文件） | timeout | 2次 |
| 并发导致 429 | 子代理首次 API 调用就被拒 | 2次 |
| context window 超限 | input tokens 超过模型上限 | 1次 |

---

## 二、根因分析

### 2.1 核心根因：共享 API 配额 + 无排队机制

**当前配置**：
- 模型：`glm-5-turbo`（智谱 GLM-5）
- API 地址：`https://open.bigmodel.cn/api/paas/v4`
- **只有一个 API Key**，所有 session（主 session + cron + 子代理）共享
- OpenClaw **没有内置的请求排队/限速机制**

**这意味着**：
- 主 session 在跟用户对话时，如果同时 spawn 子代理，两者竞争同一个 API 配额
- cron 每4分钟触发一次，如果上一次的 session 还在运行，就是两个请求同时发出
- 同时 spawn N 个子代理 = 同时发 N+1 个请求（主 session + N 子代理）

**429 的本质**：智谱 API 的每分钟请求数（RPM）或每分钟 token 数（TPM）被超了。当多个 session 并发请求时，很容易触顶。

### 2.2 子代理失败的三个子原因

**原因A：429 连锁失败**
- 子代理启动后的第一次 API 调用就触发 429
- 子代理没有重试机制（或重试次数不够）
- 结果：直接标记为 failed

**原因B：任务过大导致 timeout**
- 子代理的默认 timeout 是 240 秒
- 如果任务需要读多个大文件（如 gstack 的 SKILL.md 每个 2000-8000 字），token 消耗大
- 子代理花大量时间读文件，来不及生成输出就超时

**原因C：context window 超限**
- 子代理在 isolated session 中运行
- 如果任务涉及读取大文件（如 progress.md 曾膨胀到165KB），input tokens 超过 200k 上限
- 直接崩溃

## 三、解决方案

### 3.1 立即可做（零成本）

#### 方案1：永远只串行 spawn 子代理 ⭐⭐⭐
**规则**：任何时候都只保持一个子代理在运行。等上一个完成后再启动下一个。

**操作**：
- 如果需要研究多个文件/目录，不要同时 spawn 多个子代理
- 改为一个子代理做完所有事，或者串行 spawn

**预期效果**：消除 90% 的 429 错误

#### 方案2：子代理任务要小而聚焦 ⭐⭐⭐
**规则**：每个子代理只做一件事，任务描述控制在 200 字以内。

**操作**：
- ❌ 不要说"读取7个 skill 文件并分析"（太大）
- ✅ 改为"读取 office-hours/SKILL.md，提取核心功能和工作流程"（小而聚焦）
- 如果需要读多个文件，让子代理自己分批读，或拆成多个串行子代理

**预期效果**：减少 timeout 和 context window 超限

#### 方案3：增加子代理 timeout ⭐⭐
**规则**：对于需要读取多个文件的任务，设置更长的 timeout。

**操作**：
- 简单任务：默认 240 秒即可
- 需要读 3+ 个文件的任务：设置 360-480 秒
- `sessions_spawn` 时指定 `runTimeoutSeconds`

#### 方案4：让子代理写文件而非返回大量文本 ⭐⭐⭐
**规则**：子代理的输出应该写入文件，不要试图把大段文本返回给主 session。

**操作**：
- 在任务描述中明确说"将结果写入 XXX 文件"
- 主 session 只读文件的关键摘要
- 这也是 AGENTS.md SOP 铁律的要求

### 3.2 需要配置（低成本）

#### 方案5：配置 cron 间隔 ≥ 任务执行时间 ⭐⭐
**规则**：cron 的间隔必须大于单个任务的最长执行时间。

**当前问题**：cron 间隔 4 分钟（240s），但 timeout 也是 240s——意味着如果任务刚好超时，下一个 cron 已经触发了，两个并行请求。

**操作**：
- 如果任务平均需要 90 秒，cron 间隔设为 5 分钟（300s）
- 如果任务平均需要 150 秒，cron 间隔设为 6 分钟（360s）
- 公式：cron 间隔 = 平均执行时间 × 1.5

#### 方案6：配置 OpenClaw 的请求限速（如果支持）
**检查**：OpenClaw 是否支持 `agents.defaults.rateLimit` 或类似配置？

**需要进一步确认**：查看 OpenClaw 文档是否有请求排队/限速的配置项。

### 3.3 需要投入（中成本）

#### 方案7：多 API Key 轮转
**思路**：申请多个智谱 API Key，配置轮转，分散单 Key 的压力。

**操作**：
- 申请 2-3 个 API Key
- 配置 OpenClaw 的 providers 使用多个 key（需要确认是否支持）
- 或者用不同的 provider alias 指向不同的 key

#### 方案8：使用不同模型做不同任务
**思路**：主 session 用 glm-5-turbo（质量优先），子代理用更轻量的模型（速度优先）。

**操作**：
- 如果有 glm-4 或其他更便宜的模型可用
- `sessions_spawn` 时指定 `model` 参数使用轻量模型
- 减少 token 消耗，降低触发 429 的概率

## 四、行动规则（写入 SOP）

基于以上分析，以下规则从现在开始执行：

### 铁律1：永远串行，禁止并发
- ❌ 禁止同时 spawn 2个以上子代理
- ✅ 一个子代理完成后再启动下一个
- ✅ 主 session 等待子代理完成后再发下一条消息

### 铁律2：子代理任务要小
- ❌ 一个子代理不要读超过3个大文件
- ✅ 任务描述控制在 200 字以内
- ✅ 结果写文件，不返回大段文本

### 铁律3：cron 间隔 > 任务时间 × 1.5
- ❌ 不要让 cron 间隔等于或小于 timeout
- ✅ 根据实际执行时间动态调整

### 铁律4：监控 429，即时调整
- 出现 429 → 立即停止所有并行操作
- 等待 60 秒后重试
- 如果连续 2 次 429 → 降低任务频率或减小任务规模

## 五、今天 gstack 研究的正确做法

按上述规则，正确的做法应该是：

**错误做法**（我刚犯的）：
```
同时 spawn 2 个子代理 → 429 → 双双失败
```

**正确做法**：
```
spawn 子代理1（batch1: 7个 skill）→ 等完成 → 读结果
spawn 子代理2（batch2: 7个 skill）→ 等完成 → 读结果
或者：
spawn 子代理1（全部14个 skill，但分批读取，结果写文件）→ 等完成 → 读文件
```
