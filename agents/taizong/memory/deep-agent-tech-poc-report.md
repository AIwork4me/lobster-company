## 一、核心技术栈调研

### 1.1 LLM 选择

| 模型 | Agent 适配度 | 核心优势 | 核心劣势 | 适合场景 |
|------|-------------|---------|---------|---------|
| **Claude (Sonnet 4/Opus 4)** | ⭐⭐⭐⭐⭐ | 最强工具调用能力；超长上下文（200K）；指令遵循精准；安全对齐好 | 价格偏高；国内访问需代理；API 稳定性偶有波动 | **复杂 Agent 底层首选**，多步推理+工具编排 |
| **GPT-4o / o3** | ⭐⭐⭐⭐⭐ | 生态最成熟；Function calling 标准制定者；多模态强 | 价格高；上下文窗口相对小；OpenAI 政策变动风险 | 需要多模态能力的 Agent；快速原型 |
| **Gemini 2.5 Pro** | ⭐⭐⭐⭐ | 1M 超长上下文；Google 生态集成；价格有竞争力 | Agent 框架支持较少；工具调用稳定性不如 Claude | 需要处理超长文档的场景 |
| **DeepSeek V3/R1** | ⭐⭐⭐⭐ | **性价比之王**；R1 推理能力强；开源可自部署 | API 稳定性一般；工具调用格式不够成熟；中文强英文稍弱 | **PoC 和成本敏感场景**；国内部署 |
| **Qwen (通义千问)** | ⭐⭐⭐ | 国内合规；开源可选；阿里云集成好 | Agent 编排能力弱于 Claude/GPT；复杂推理有差距 | 国内企业客户；合规要求高的场景 |
| **Llama 3 (开源)** | ⭐⭐⭐ | 完全可控；可微调；无 API 成本 | 需要强 GPU 集群；能力弱于闭源模型 | 极端自主需求；私有化部署 |

**Cherny 的判断：**
- **Agent 核心推理层**：Claude Sonnet 4 是当前最强选择，工具调用+多步推理综合能力第一
- **性价比选择**：DeepSeek R1 for reasoning tasks, V3 for general tasks
- **务必保持多模型支持**：不绑死单一供应商，用抽象层隔离

### 1.2 Agent 框架

| 框架 | 定位 | 优势 | 劣势 | 适合场景 |
|------|------|------|------|---------|
| **LangChain / LangGraph** | 通用 Agent 框架 | 生态最大；工具集成多；LangGraph 提供状态机式工作流 | 抽象层太厚；调试困难；版本变动频繁；"胶水代码"多 | 快速原型；需要大量第三方集成 |
| **LlamaIndex** | RAG + 知识 Agent | 数据连接器丰富；RAG 链路成熟 | Agent 编排能力不如 LangGraph | 知识密集型 Agent |
| **CrewAI** | 多 Agent 协作 | 多角色编排直觉好；上手快 | 单 Agent 能力弱；定制性受限 | 多角色协作场景（如研究团队模拟） |
| **AutoGen (Microsoft)** | 多 Agent 对话 | 微软背书；研究导向 | 生产化不够成熟 | 研究/实验 |
| **OpenAI Agents SDK** | 官方 Agent 框架 | 与 GPT 深度集成；简洁 | 仅支持 OpenAI 模型 | 纯 OpenAI 生态 |
| **Pydantic AI** | 结构化输出 Agent | 类型安全；代码质量高 | 生态较小；功能较少 | 需要强类型保障的 Agent |
| **Vercel AI SDK** | 前端+Agent | 流式体验好；前后端一体 | 后端能力有限 | 面向消费者的 Web Agent 产品 |

**Cherny 的判断：**
- **不建议直接用任何框架作为核心**——它们都是"脚手架"，不是"引擎"
- **LangGraph 的状态机思想值得借鉴**，但不要依赖其抽象
- **最务实的路径**：自建轻量级 Agent Loop（核心就几百行代码），必要时按需集成具体工具（如用 MCP 做工具层）
- **如果一定要选一个快速起步**：LangGraph（生态好、生产案例多），但长期必须脱离

### 1.3 记忆系统

| 方案 | 类型 | 优势 | 劣势 | 成熟度 |
|------|------|------|------|--------|
| **Mem0** | 托管/开源记忆层 | 接入简单（一行代码）；自动提取记忆；100K+ 开发者使用 | 核心数据托管在第三方；定制性有限；深度推理能力弱 | ⭐⭐⭐⭐ 生产可用 |
| **Letta (MemGPT)** | 开源 Agent+记忆框架 | 学术背景强（UC Berkeley）；self-editing memory；agent 可自主管理记忆 | 架构复杂；学习曲线陡；版本迭代快不够稳定 | ⭐⭐⭐ 快速发展中 |
| **自建方案** | 定制 | 完全可控；可针对场景优化 | 开发成本高；需要大量验证 | 取决于投入 |

**记忆系统架构拆解（Cherny 的分析）：**

真正有效的 Agent 记忆需要三层：

1. **工作记忆（Working Memory）**：当前任务的上下文窗口 → **LLM 本身就是**
2. **短期记忆（Short-term）**：当前会话的关键信息 → 用 structured JSON + embedding 存储，RAG 检索
3. **长期记忆（Long-term）**：跨会话的用户画像、经验教训、技能 → 向量数据库 + 知识图谱

**Cherny 的判断：**
- **PoC 阶段**：用 Mem0 快速验证记忆价值（不重复造轮子）
- **产品阶段**：必须自建记忆引擎——这是 Deep Agent 的核心竞争力之一
- **Letta 的 self-editing memory 思想非常值得学习**，但它的实现还不够生产级
- 记忆的关键不是存储，而是**什么时候记、记什么、什么时候忘、怎么检索**——这才是壁垒

### 1.4 工具调用现状

**Function Calling（各大模型原生支持）：**
- OpenAI、Anthropic、Google 都已原生支持
- 标准格式：JSON Schema 描述工具 → 模型输出结构化调用 → 应用执行 → 结果返回
- **问题**：每个厂商格式不同，需要适配层

**MCP（Model Context Protocol，Anthropic 主导）：**
- 本质：**AI 应用的 USB-C 标准**——统一的工具发现和调用协议
- 架构：Client（AI 应用）↔ Server（工具提供方），通过标准化的 JSON-RPC 通信
- 生态：Claude、ChatGPT、VS Code、Cursor 等已支持；MCP Server 社区快速增长
- **优势**：写一次工具，所有 MCP Client 都能用
- **问题**：协议还在快速迭代；性能开销（多一层抽象）；安全模型还在完善

**Cherny 的判断：**
- **MCP 是正确的方向**——工具生态的标准化是必然趋势
- **PoC 阶段**：同时支持 Function Calling（直连模型）和 MCP（连接工具生态）
- **长期**：以 MCP 为工具层标准，但在核心 Agent Loop 内部保持轻量级自定义 tool calling
- **注意**：MCP 是工具协议，不是 Agent 框架——不要指望 MCP 解决 Agent 的推理和规划问题

## 二、技术架构草案

### 2.1 整体架构（分层描述）

```
┌─────────────────────────────────────────────────┐
│              用户界面层 (Presentation)            │
│  Web UI / CLI / API Gateway / Mobile SDK         │
├─────────────────────────────────────────────────┤

│  会话路由 / 上下文管理 / 权限控制 / 流式输出       │

│  ┌───────────────────────────────────────────┐   │
│  │  推理引擎 (Reasoning Engine)              │   │
│  │  - 任务分解 / 多步推理 / 反思 / 自我纠正    │   │
│  ├───────────────────────────────────────────┤   │

│  ├───────────────────────────────────────────┤   │
│  │  记忆引擎 (Memory Engine) ← 核心壁垒       │   │
│  │  - 记忆提取 / 存储 / 检索 / 遗忘 / 压缩    │   │
│  ├───────────────────────────────────────────┤   │

│  ├───────────────────────────────────────────┤   │
│  │  进化引擎 (Evolution Engine) ← 长期壁垒    │   │
│  │  - 经验固化 / 技能习得 / 自我改进           │   │
│  └───────────────────────────────────────────┘   │
├─────────────────────────────────────────────────┤
│              模型抽象层 (Model Abstraction)       │
│  LLM Router / Fallback / Cost Optimizer          │
│  (Claude / GPT / DeepSeek / 开源模型)            │
├─────────────────────────────────────────────────┤
│              工具层 (Tools)                       │

│  (文件系统 / 代码执行 / 搜索 / 浏览器 / API)     │
├─────────────────────────────────────────────────┤

│  向量数据库 / 知识图谱 / 关系数据库 / 对象存储     │
└─────────────────────────────────────────────────┘
```

### 2.2 各模块职责

**推理引擎（Reasoning Engine）：**
- 接收用户任务，生成推理链（Chain of Thought）
- 实现反思机制：完成一步后自我评估，发现错误则回溯
- 关键技术：Structured Output（Pydantic 模型）、Reflection Pattern

**规划引擎（Planning Engine）：**
- 将复杂目标分解为可执行的子任务 DAG
- 执行过程中根据反馈动态调整计划
- 关键技术：Plan-and-Execute、ReAct、动态优先级

**记忆引擎（Memory Engine）：**
- 三层记忆：工作记忆（LLM context）→ 短期记忆（会话级）→ 长期记忆（跨会话）
- 自动提取、压缩、检索、遗忘
- 关键技术：Embedding + Vector DB、Memory Compression、Recency/Relevance/Importance 评分

**工具编排（Tool Orchestration）：**
- MCP 协议连接外部工具生态
- 工具选择策略：基于任务上下文和工具描述的语义匹配
- 错误恢复：工具调用失败时的自动降级和重试

**进化引擎（Evolution Engine）：**
- 从成功/失败经验中提取教训
- 将教训转化为可复用的 SOP 和 prompt 技能
- 关键技术：经验回放、技能库、Prompt 优化

### 2.3 关键技术难点

| 难点 | 说明 | 应对方案 |
|------|------|---------|
| **长任务中的上下文丢失** | 多步任务执行到后期，早期上下文被 LLM "遗忘" | 分层记忆 + 关键信息主动注入；任务状态机持久化 |
| **工具调用的可靠性** | LLM 生成错误的工具参数、调用不存在的工具 | 结构化输出（JSON Schema 强制校验）；工具调用验证层 |
| **计划质量的不可控性** | LLM 生成的计划可能不完整、有遗漏 | 多次规划取最优；人类审批关键节点；反思机制 |
| **记忆检索的精准度** | 检索到的记忆不相关，或遗漏关键记忆 | Hybrid Search（向量+关键词）；Re-ranking；多轮检索 |
| **成本控制** | 复杂任务可能消耗大量 Token | 轻量模型做简单任务；缓存机制；Token 预算管理 |
| **安全性** | Agent 执行危险操作（删数据、发邮件） | 权限沙箱；操作审批；危险等级分级 |

## 三、PoC 方案

### 3.1 PoC 目标

**用最少的代码，验证一个核心假设：自研 Agent Loop + 记忆系统，能否比直接调 LLM API 产生明显更好的复杂任务完成率。**

### 3.2 核心假设验证

1. **记忆假设**：带记忆的 Agent 比无记忆的 Agent，在多轮复杂任务中完成率高 30%+
2. **推理假设**：结构化推理（Plan → Execute → Reflect）比单次调用，任务完成率高 40%+
3. **工具假设**：MCP 工具编排能显著扩展 Agent 的能力边界

### 3.3 技术实现路径

**语言**：Python（生态最成熟，Agent 框架/LLM SDK 最丰富）
**模型**：DeepSeek V3（便宜，够用）+ Claude Sonnet 4（对比基准）
**核心代码量**：预计 500-800 行核心代码（不含工具集成）

```
技术栈：
- LLM SDK: anthropic SDK + openai SDK（兼容 DeepSeek API）
- Agent Loop: 自建（Plan-Execute-Reflect 循环）
- 记忆: Mem0（快速验证）+ 本地 SQLite（备选）
- 工具: MCP 协议 + 自定义 Function Calling
- 编排: 简单的状态机（Python dict + enum）
- 测试: 自动化评测脚本（对比有/无记忆、有/无反思）
```

### 3.4 PoC 架构（精简版）

```python
# 伪代码 - 核心循环不超过 200 行

class DeepAgent:
    def __init__(self, llm, memory, tools):
        self.llm = llm
        self.memory = memory
        self.tools = tools
    
    async def run(self, task: str) -> str:
        # 1. 检索相关记忆
        context = await self.memory.search(task)
        
        # 2. 生成计划
        plan = await self.llm.plan(task, context)
        
        # 3. 逐步执行
        for step in plan.steps:
            # 执行工具调用或推理
            result = await self.execute_step(step)
            
            # 反思：结果是否符合预期？
            reflection = await self.llm.reflect(step, result)
            if reflection.needs_revision:
                # 自我纠正
                result = await self.revise(step, reflection)
```

### 3.5 评测任务设计

PoC 必须有**可量化的评测**，不能用"感觉"判断好坏：

| 任务类型 | 示例 | 评测标准 |
|---------|------|---------|
| 多步研究 | "调研 MCP 协议的现状，输出 1000 字报告" | 完整性、准确性（人工评分） |
| 多工具协作 | "读取这个文件 → 分析数据 → 生成图表 → 发送邮件" | 步骤完成率、工具调用正确率 |
| 记忆测试 | 第一次告知偏好，第二轮任务中体现 | 偏好利用率 |
| 错误恢复 | 给一个会失败的工具调用，看 Agent 能否恢复 | 恢复成功率 |

## 四、独立自主评估

### 4.1 不做套壳的难点（实事求是）

| 难点 | 难度 | 说明 |
|------|------|------|
| **记忆系统** | 🔴 高 | 不是存取数据这么简单。什么时候记、记什么、怎么关联、怎么遗忘——每个决策都需要大量实验验证 |
| **推理质量** | 🔴 高 | Plan-Execute-Reflect 的 prompt engineering 是"玄学"——需要大量 A/B 测试 |
| **工具可靠性** | 🟡 中 | LLM 生成 JSON 参数的稳定性不够，需要多层校验和重试机制 |
| **评测体系** | 🟡 中 | 没有标准化的 Agent 评测基准，需要自建 |

### 4.2 开源 vs 自研

| 组件 | 开源可用 | 自研部分 | 原因 |
|------|---------|---------|------|
| LLM 推理 | ✅ 全部用 API | 模型路由层（薄层封装） | 自训模型成本太高，暂不现实 |
| Agent Loop | ❌ | **核心自研** | 这是灵魂，不能抄 |
| 记忆引擎 | ⚠️ Mem0 作为起点 | 记忆策略（提取/检索/遗忘算法） | 存储层用开源，策略层自研 |
| 工具层 | ✅ MCP 协议 | 工具选择策略 | MCP 是标准，站在巨人肩膀上 |
| 安全沙箱 | ✅ Docker/e2b | 权限模型设计 | 容器用开源，策略自研 |
| 评测框架 | ⚠️ 参考开源 | 评测任务设计 | 场景自建，框架可借鉴 |
| 前端 | ✅ | 交互设计 | 工程实现，非核心 |

### 4.3 技术壁垒如何建立

**短期（0-6个月）壁垒：**
- Agent Loop 的 prompt 工程积累（Plan-Execute-Reflect 的具体实现）
- 记忆策略的实验数据（什么该记、什么该忘，需要大量真实场景打磨）

**中期（6-18个月）壁垒：**
- 进化引擎——Agent 从经验中学习和改进的能力（这是最难的，也是最值钱的）
- 评测数据集——大量真实场景下的任务完成率数据，形成飞轮效应

**长期壁垒：**
- 记忆和进化能力形成的数据飞轮：用得越多，Agent 越聪明，用户越离不开
- 垂直领域的深度优化（比如针对编程、研究、运营等场景的专门调优）

**诚实的评估：**
- 纯技术壁垒在 AI 时代很脆弱——prompt 和策略可以被逆向
- **真正的壁垒是数据飞轮 + 用户生态**——技术是手段，不是目的
- 但如果不掌握核心技术，连参与竞争的资格都没有

## 五、结论（3 句话）

1. **技术路线推荐**：自研 Agent Loop + 记忆引擎为核心（这是灵魂），LLM 用 DeepSeek V3 控成本 + Claude Sonnet 4 保质量，工具层基于 MCP 协议，PoC 阶段用 Mem0 做记忆层快速验证。

2. **核心风险**：推理质量和记忆策略是两个最大不确定性——LLM 的推理能力天花板决定了 Agent 的上限，而记忆系统的"该记什么、该忘什么"目前没有成熟方案，需要大量实验；此外，成本控制是长期挑战，复杂任务的 Token 消耗可能远超预期。

3. **PoC 需要多久**：一个 1-2 人工程师团队，8-12 天可以完成最小 PoC（自建 Agent Loop + Mem0 记忆 + MCP 工具 + 自动化评测），验证"有记忆+有反思的 Agent 比裸调 LLM API 好多少"这个核心假设——如果 2 周内验证不通过，立即转向，不恋战。
