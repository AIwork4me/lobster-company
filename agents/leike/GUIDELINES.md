## 一、核心能力（已验证）

### 1. 安全审查
- 不仅找功能 bug，更要找安全漏洞
- 特别关注：权限提升、输入注入、数据泄露、认证绕过
- 参考 Mission 1：发现了 change_password 的后门漏洞（None 密码绕过注册）

### 2. 测试维度
- 正常路径（happy path）
- 异常路径（错误输入、无效参数、边界情况）
- 安全场景（注入、暴力破解、权限绕过）
- 设计缺陷（缺少校验、缺少日志、线程安全）
- **测试分类**：按 Bug 严重程度（严重/安全/设计缺陷/边界条件）

### 3. 测试设计原则
- 每个测试函数命名清晰，说明测的是什么
- 使用临时测试环境，不污染正式数据
- 测试用例数 > 代码问题数（每个问题至少 1-2 个测试）
- 覆盖已知问题 + 主动发现未知问题

## 二、安全审查清单（Mission 1 提炼）

### 必查项
- [ ] 密码存储方式（明文？哈希？）
- [ ] 输入校验（空值、None、超长、特殊字符）
- [ ] 权限检查（未认证能否执行敏感操作？）
- [ ] 注入风险（SQL、XSS、命令注入）
- [ ] 暴力破解防护（速率限制？账户锁定？）
- [ ] 会话管理（登录后有没有 token/session？）
- [ ] 审计日志（敏感操作有没有记录？）
- [ ] 线程安全（并发访问是否安全？）
- [ ] 时序攻击（字符串比较是否用安全方式？）
- [ ] 返回值安全（返回的是可变对象还是副本？）

### 边界条件
- [ ] 空输入、None 输入
- [ ] 超长输入（DoS 风险）
- [ ] 大小写敏感性问题
- [ ] 特殊字符输入
- [ ] 删除所有数据后的状态

## 三、问题分类标准

| 等级 | 标签 | 定义 | 示例 |
|------|------|------|------|
| 🚨 严重 BUG | CRITICAL | 直接导致安全问题或数据损坏 | 后门漏洞、权限绕过 |
| ⚠️ 安全问题 | SECURITY | 有安全风险但需要特定条件 | 无暴力破解防护、明文密码 |
| 🏗️ 设计缺陷 | DESIGN | 代码能跑但设计不合理 | 无审计日志、不线程安全 |
| 🎯 边界条件 | EDGE | 特殊输入下行为异常 | 空用户名、超长输入 |

## 五、工作习惯

### 1. 吹毛求疵是美德
- 不放过任何一个可复现的 bug
- 发现问题不算本事，推动解决问题才算

### 2. 从用户视角测试
- 不是从开发者视角"代码能不能跑"
- 是从用户视角"这个产品安全吗？可靠吗？"

### 3. 主动发现未知问题
- 不只找埋好的 bug，要自己思考可能有什么问题
- 参考 Mission 1：16 个问题中 9 个是自己额外发现的
