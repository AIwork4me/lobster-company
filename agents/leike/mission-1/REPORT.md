## 一、问题清单（共发现 16 个问题）

### 🚨 严重 BUG（3 个）

| # | 问题 | 位置 | 说明 |
|---|------|------|------|
| 1 | **change_password 后门：None 旧密码可创建任意用户** | `change_password()` | `users.get(username)` 对不存在的用户返回 None，如果 `old_password` 也是 None，条件 `!= None` 为 False，直接跳到 `self.users[username] = new_password`，等于绕过注册直接创建用户。**这是后门级的漏洞。** |
| 2 | **密码明文存储** | 全局设计 | 所有密码以明文存入字典，任何能访问 `auth.users` 的代码都能读取全部密码。内存转储、序列化、日志打印都会泄露。 |
| 3 | **无输入校验：允许 None/空字符串/任意类型作为用户名和密码** | `register()` | `""`、`None`、数字、元组等都能成为合法用户名，空密码被接受。 |

### ⚠️ 安全问题（4 个）

| # | 问题 | 说明 |
|---|------|------|
| 4 | **无暴力破解防护** | login 没有速率限制、没有账户锁定，攻击者可无限次尝试 |
| 5 | **明文密码比较（时序攻击风险）** | 使用 `==` 比较密码字符串，理论上可通过响应时间推断正确密码前缀 |
| 6 | **无输入清洗** | 用户名可以包含 SQL 注入字符串、XSS payload 等，虽然当前是纯内存字典无直接危害，但数据迁移到数据库时会引爆 |
| 7 | **无会话管理** | login 只返回静态字典，没有 token、session、过期时间，无法实现真正的认证状态追踪 |

### 🏗️ 设计缺陷（6 个）

| # | 问题 | 说明 |
|---|------|------|
| 8 | **无密码强度校验** | 允许空密码、单字符密码、新密码与旧密码相同 |
| 9 | **无用户名/密码长度限制** | 超长输入（10000 字符）被直接接受，可能被用于 DoS |
| 10 | **用户名大小写敏感** | `Alice` 和 `alice` 是两个不同用户，用户体验混乱 |
| 11 | **纯空格用户名被允许** | `"   "` 作为用户名被接受，极容易误操作 |
| 12 | **无审计日志** | 注册、登录、修改密码、删除用户等敏感操作没有任何日志记录 |
| 13 | **非线程安全** | 多线程并发操作没有同步机制，可能导致数据竞争 |
| 14 | **login 返回可变字典** | 返回的字典可以被外部代码修改，没有防御性拷贝 |

## 三、最严重的问题

**🚨 #1: change_password 后门漏洞**

```python
def change_password(self, username, old_password, new_password):
    if self.users.get(username) != old_password:
        return False
    self.users[username] = new_password
    return True
```

**为什么最严重：**
- 任何人调用 `change_password("任意用户名", None, "我设的密码")` 就可以**绕过注册直接创建任意用户**
- 这是一个**权限提升漏洞**：未认证的用户可以创建账户并设置密码
- 不需要知道任何已有用户的凭据
- 攻击面极大，一行代码就能触发

**修复建议：** 增加 `username in self.users` 的前置检查。
