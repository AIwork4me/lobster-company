## 一、技术偏好（从实战中验证，不是假设）

### 1. 优先使用 Python 标准库
- 零外部依赖是第一优先级
- 网络请求用 `urllib.request`，不用 requests
- HTML 解析用 `html.parser.HTMLParser`，不用 BeautifulSoup
- CLI 参数用 `argparse`
- 测试用 `unittest`

### 2. 用 AST 解析，不用正则
- Python 代码分析必须用 `ast` 模块
- 正则只能用于简单的字符串匹配（URL验证、标题提取等）
- AST 能获取结构信息（函数定义、嵌套深度、类结构），正则做不到

### 3. 模块化拆分，职责单一
- 不要把所有功能塞进一个文件
- 按职责拆分模块：统计、分析、报告、CLI、核心逻辑等
- 每个文件一个明确职责，超过 300 行考虑拆分
- 用 `__init__.py` 组织为 Python 包

## 二、代码质量标准（自检清单）

### 每次提交前检查：
- [ ] 函数行数 ≤ 50 行（超过就要拆分）
- [ ] 函数参数 ≤ 6 个（超过考虑用 dataclass 或 dict）
- [ ] 嵌套深度 ≤ 4 层（超过就重构，用 early return / 提取子函数）
- [ ] 空的 except 块不允许（至少 `pass` + 注释说明原因）
- [ ] 不使用 `eval()` 和 `exec()`
- [ ] 不使用 `from xxx import *`（通配符导入）
- [ ] 文件有 docstring（模块级别）
- [ ] 复杂度高的函数（>10）必须有注释说明

## 三、测试标准（不可妥协）

### 必须写测试，没有例外
- 测试用 `unittest`，放在 `tests/` 目录
- 测试使用临时目录，不污染正式数据
- 必须覆盖：
  - ✅ 正常路径（happy path）
  - ✅ 异常路径（错误输入、文件不存在、网络失败）
  - ✅ 边界条件（空输入、超大输入、特殊字符）
  - ✅ 数据完整性（添加→读取→删除的完整生命周期）
- 测试函数名要有意义：`test_search_keyword` 比 `test_search_1` 好
- **必须运行测试并确认全部通过，才算任务完成**

### 参考 Mission 2 的测试覆盖：
- 16 个测试用例，覆盖 5 个核心场景
- 用 `tempfile.mkdtemp()` 隔离测试环境
- 测试时间应该 < 1 秒（轻量快速）

### 参考 Mission 3 的测试覆盖：
- 34 个测试用例，覆盖 6 个维度
- 分成多个 TestCase 类，按功能模块组织
- 包含集成测试（真实文件分析）

## 四、已知弱点（你自己的工具检测出来的，要特别注意）

- **对策**：写完代码后，目测检查是否有相似的代码块，提取为公共函数

## 五、工作习惯

### 1. 先理解需求，再动手编码
- 不要看到任务就立刻开始写代码
- 先想清楚：用户要什么？什么算做好了？有没有歧义？

### 2. 自我分析
- 如果可能，用工具分析自己写的代码
- 发现问题不隐藏，诚实报告
- 参考 Mission 3：Cherny 给自己的 checks.py 打了 53 分

### 3. 环境适配
- 当前环境是 Windows 11 + PowerShell
- PowerShell 不支持 `&&`，用 `;` 分隔命令
- 创建目录用 `New-Item -ItemType Directory -Force`，不用 `mkdir -p`
- 注意 Windows 文件编码（UTF-8 with BOM 可能出问题）

### 4. 任务完成后必须交付
- 功能代码
- 测试代码 + 测试通过截图
- 任务完成报告（做了什么、遇到什么问题、自我评价）
