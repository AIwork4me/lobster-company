> 目标：在推进团队协作之前，确保 Cherny 是一个真正"培养好"的 AI 工程师。
> 原则：多算胜，少算不胜——先全局规划，再逐步执行。

## 二、完整实践计划（4个阶段，8个任务）

### Phase 1：育人验证（2个任务）

验证 GUIDELINES.md 是否真的能让他做得更好。

**Mission 4：育人闭环验证**
- 目标：验证他是否自动遵守 GUIDELINES 中的经验
- 方法：给一个类似 Mission 3 的任务，但**任务描述里不提任何技术偏好**
- 验证点：
  - 他是否自动用 AST 而不是正则？
  - 他是否自动写测试？
  - 他的函数是否自动控制在 50 行以内？
  - 嵌套深度是否自动控制在 4 层以内？
- 预期：如果育人生效，他不需要我提醒就能自觉遵守标准
- **如果失败**：说明 GUIDELINES 还不够强，需要调整格式或内容

**Mission 5：迭代改进**
- 目标：验证他能不能在反馈后改进代码
- 方法：
  1. 给一个任务，让他完成第一版
  2. 我用他 Mission 3 的代码质量分析器扫描他的代码
  3. 把扫描结果（问题列表）给他
  4. 让他修复所有问题，交付第二版
  5. 再扫描，看是否改进
- 验证点：
  - 他能理解别人的反馈吗？
  - 修复是"治标"还是"治本"？
  - 第二版评分是否明显提升？
- 这是协作的前置能力——团队协作中他一定会收到别人的反馈

### Phase 2：进阶能力（2个任务）

**Mission 6：模糊需求 + TDD**
- 目标：同时验证两个能力
- 方法：给一个**故意模糊**的需求（比如"做一个帮助开发者更高效的东西"），不指定技术方案
- 验证点：
  - 他会先提问澄清需求吗？（还是直接动手写？）
  - 他在编码前会先写测试吗？（TDD）
  - 他最终做出的东西是否合理？
- 这模拟的是真实场景：产品经理的需求往往不完美

**Mission 7：代码审查**
- 目标：验证他能发现问题、给出有价值的建议
- 方法：给他一段**故意有问题的代码**（可能是他自己的 Mission 3 代码，也可能是别人写的）
- 验证点：
  - 他能发现明显bug吗？
  - 他能发现设计缺陷吗？
  - 他的 review 意见具体可操作吗？（不是"这里不太好"这种废话）
- 这为团队协作做准备——他需要能审查 Chase 或其他人写的代码

### Phase 3：高级能力（2个任务）

**Mission 8：架构设计**
- 目标：验证他在编码前能先出设计方案
- 方法：给一个需要**架构权衡**的复杂需求（比如"一个插件系统，支持动态加载和卸载"）
- 要求：先出设计文档（接口定义、模块关系、数据流），经我审批后再编码
- 验证点：
  - 设计文档是否清晰完整？
  - 有没有考虑边界情况和扩展性？
  - 审批后的实现是否和设计一致？
- 这是与 Steinberger（架构师）配合的前置能力

**Mission 9：读需求文档并实现**
- 目标：验证他能理解别人写的文档
- 方法：由我（扮演产品经理）写一份需求文档，让他阅读后实现
- 验证点：
  - 他有没有遗漏需求中的关键点？
  - 他有没有发现需求中的歧义并提问？
  - 实现结果是否和需求文档一致？
- 这是团队协作的核心能力

### Phase 4：综合验证（2个任务）

**Mission 10：综合战场**
- 目标：在一个人身上模拟完整的开发流程
- 方法：给一个中等复杂度的任务，要求完整流程：
  1. 先出设计方案
  2. 写测试（TDD）
  3. 实现
  4. 自我审查（用自己的工具）
  5. 修复问题
  6. 写技术文档
  7. 交付报告
- 验证点：全流程的质量是否都达标？
- 这是"毕业考试"——通过了说明 Cherny 已经是一个完全培养好的工程师

**Mission 11：极限压力测试**
- 目标：给他一个真正有难度的任务
- 方法：一个需要多种技术能力组合的任务（具体待定，基于前10个任务的暴露情况）
- 验证点：在压力下他是否还能保持质量？
- 这是"压力测试"——确认能力上限

## 三、验收机制

### 每个任务的标准流程

```
1. 我写任务描述（只写目标和验收标准，不写实现方式）
2. spawn Cherny 执行
3. 我亲自验收：
   - 运行代码
   - 运行测试
   - 用代码质量分析器扫描
   - 检查交付物完整性
4. 评分（A/B/C/D）
5. 更新 GUIDELINES.md（新经验追加）
6. 更新实战项目档案
7. 写当日日志
```

### 评分标准

| 等级 | 含义 |
|------|------|
| A | 超出预期，有我没想到的好决策 |
| B | 达标，符合验收标准 |
| C | 基本达标，但有明显瑕疵 |
| D | 不达标，需要重做 |

### 红线（出现即重做）
- 测试不通过
- 代码无法运行
- 没有任务完成报告
- 没有写测试

## 四、时间规划

| 阶段 | 任务数 | 预计耗时 | 关键里程碑 |
|------|--------|---------|-----------|
| Phase 1 | 2 | 1-2天 | 育人闭环验证通过 |
| Phase 2 | 2 | 1-2天 | 能处理模糊需求、能做代码审查 |
| Phase 3 | 2 | 1-2天 | 能出设计方案、能读需求文档实现 |
| Phase 4 | 2 | 1天 | 综合毕业考试通过 |

**总计：8个任务，预计4-6天**

但节奏不是固定的——如果某个阶段暴露了严重问题，停下来解决再继续。"多算胜"不是"死板执行"。

## 五、已知风险

1. **Rate Limit（429）** — 三次都遇到，仍是最大障碍。每次任务间隔 > 10分钟
2. **Session 超时** — 复杂任务可能超过 600 秒限制。备选：拆分成子任务
3. **育人可能不生效** — 如果 Mission 4 显示 GUIDELINES 没用，需要重新设计"教育"方式
4. **上下文限制** — 任务描述太长会占 token。保持简洁，只写目标和标准
