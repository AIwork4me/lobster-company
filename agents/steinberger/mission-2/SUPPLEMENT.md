## 补充一：全局错误处理与恢复策略（对应 Mission 1 扣分 -8）

### 1.1 问题回顾

Mission 1 的 Reflection Engine 只处理"Agent 行动结果质量不高"这一类问题（quality_score 低 → 重试/换策略）。但实际运行中会面临大量非质量类错误：LLM 调用超时、工具执行崩溃、上下文溢出、配置缺失等。这些错误不归 Reflection Engine 管，散落在各模块内部，没有统一的分类和恢复机制。

### 1.2 错误分类体系

```
┌─────────────────────────────────────────────────────────────────┐
│                     错误分类体系                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │  Level 0: 瞬态可恢复 (Transient Recoverable)              │   │
│  │  ─────────────────────────────────────────────────        │   │
│  │  · LLM API 速率限制 (429)                                 │   │
│  │  · LLM API 临时超时 (503)                                 │   │
│  │  · 工具执行超时（资源瞬时不可用）                           │   │
│  │  恢复策略：自动重试（指数退避）                              │   │
│  │  最大重试：3 次，退避间隔 1s → 2s → 4s                     │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │  Level 1: 结构可恢复 (Structural Recoverable)              │   │
│  │  ─────────────────────────────────────────────────        │   │
│  │  · 工具参数验证失败（LLM 给了错误参数）                     │   │
│  │  · 工具返回逻辑错误（操作不合法，如删除不存在的文件）        │   │
│  │  · LLM 返回格式异常（无法解析为 ToolCall）                  │   │
│  │  恢复策略：修正后重试                                       │   │
│  │    → 参数错误 → Reflection Engine 的 rephrase 策略          │   │
│  │    → 格式异常 → Agent Core 重新请求 LLM（附格式约束）       │   │
│  │  最大重试：2 次                                            │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │  Level 2: 部分可恢复 (Partially Recoverable)                │   │
│  │  ─────────────────────────────────────────────────        │   │
│  │  · 单个子任务彻底失败（工具不存在、权限不足）                │   │
│  │  · Token 预算耗尽（但已产出部分结果）                       │   │
│  │  · 外部服务长期不可用                                      │   │
│  │  恢复策略：降级或跳过                                       │   │
│  │    → 子任务失败 → Orchestrator replan，跳过或替换           │   │
│  │    → Token 耗尽 → 压缩上下文继续 / 返回已完成的子结果       │   │
│  │    → 外部不可用 → 降级到备用工具 / 报告已完成部分           │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │  Level 3: 不可恢复 (Non-Recoverable)                       │   │
│  │  ─────────────────────────────────────────────────        │   │
│  │  · 系统提示注入导致 Agent 行为异常                          │   │
│  │  · 用户意图不明确且多轮澄清后仍无法理解                     │   │
│  │  · 硬件/基础设施故障（磁盘满、内存不足）                    │   │
│  │  恢复策略：停止执行，向用户报告错误                          │   │
│  │    → 包含错误分类、已完成的子任务结果、建议的下一步          │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 1.3 错误传播路径

```
┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
│ Tool     │     │ Agent    │     │Orchestr. │     │ I/O      │
│ execute()│────►│ Core     │────►│          │────►│ Layer    │────► 用户
│          │     │          │     │          │     │          │
│ 抛出     │     │ 捕获并   │     │ 决定     │     │ 格式化   │
│ Exception│     │ 分类     │     │ 跳过/    │     │ 为用户   │
│          │     │          │     │ replan   │     │ 可读     │
└──────────┘     └────┬─────┘     └──────────┘     └──────────┘
                      │
                      ▼
               ┌──────────────┐
               │ Error        │
               │ Classifier   │  统一错误分类
               │              │  L0→自动重试
               │              │  L1→修正重试
               │              │  L2→降级
               │              │  L3→上报
               └──────┬───────┘
                      │
                      ▼
               ┌──────────────┐
               │ Reflection   │  L1 错误的"修正后重试"
               │ Engine       │  通过 Reflection Engine
               │              │  的 rephrase 策略实现
               └──────────────┘
```

**关键设计**：引入一个轻量的 `ErrorClassifier` 组件，不作为独立大模块，而是嵌入 Agent Core 内部。它做三件事：

1. **分类**：将 Exception 映射到 Level 0-3
2. **决策**：根据 Level 决定是重试、修正、降级还是上报
3. **上报**：将不可恢复错误包装为 `UserFacingError`，交给 Orchestrator / I/O Layer

### 1.4 熔断机制

```
┌────────────────────────────────────────────────────────────┐
│                    熔断器 (CircuitBreaker)                   │
├────────────────────────────────────────────────────────────┤
│                                                             │
│  状态机：CLOSED ──(连续失败≥N)──► OPEN ──(冷却超时)──►     │
│           ▲                                          HALF  │
│           │                                          OPEN  │
│           └──────(探测成功)──────────────────────────────┘ │
│                                                             │
│  ┌───────────┐    ┌───────────┐    ┌───────────┐           │
│  │  CLOSED    │    │   OPEN    │    │HALF_OPEN  │           │
│  │           │    │           │    │           │           │
│  │ 正常放行   │    │ 全部拒绝   │    │ 放行 1 个  │           │
│  │ 失败计数   │    │ 返回降级   │    │ 探测请求   │           │
│  │ 成功→归零  │    │ 响应       │    │ 成功→CLOSE │           │
│  └───────────┘    └───────────┘    │ 失败→OPEN  │           │
│                                     └───────────┘           │
│                                                             │
│  配置（可调）：                                              │
│  · failure_threshold = 5    连续失败多少次触发熔断            │
│  · cooldown_seconds = 60   熔断后多久进入探测                │
│  · half_open_max = 1       探测时放行几个请求                │
│                                                             │
│  作用域：每个 LLM Provider / 每个外部工具 独立熔断器         │
│                                                             │
└────────────────────────────────────────────────────────────┘
```

**熔断器的降级响应**：

| 被熔断的组件 | 降级行为 |
|-------------|---------|
| LLM Provider | 切换到备用 Provider（如果配置了），否则返回"服务暂时不可用" |
| 外部工具 | 标记该工具不可用，Orchestrator replan 时排除此工具 |

### 1.5 与 Reflection Engine 的关系

```
┌──────────────────────────────────────────────────────────┐
│              两个"纠错"机制的职责边界                       │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  ErrorClassifier（错误处理）                              │
│  ─────────────────────                                   │
│  处理的是：Exception、错误码、超时                         │
│  关注的是：系统层面能不能执行                               │
│  回答的问题："这个操作能不能跑通？"                         │
│  典型场景：API 429、工具崩溃、格式解析失败                  │
│                                                          │
│  Reflection Engine（质量评估）                             │
│  ─────────────────────                                   │
│  处理的是：成功执行但结果不满意                             │
│  关注的是：语义层面结果对不对                               │
│  回答的问题："这个结果够不够好？"                           │
│  典型场景：代码生成了但有 bug、分析不完整、方向偏了          │
│                                                          │
│  交互方式：                                               │
│  ─────────                                               │
│  ErrorClassifier 处理 L1 错误时，会把修正建议传给           │
│  Reflection Engine，由后者生成 rephrase 策略。              │
│  但两者是独立的：                                          │
│  · ErrorClassifier 不做质量评估                            │
│  · Reflection Engine 不处理 Exception                     │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### ADR-006: 错误处理采用分层分类而非集中式异常处理

- **背景**：Agent 系统的错误来源多样（LLM、工具、编排、I/O），需要统一的错误管理策略。
- **选项**：
  1. 集中式 try-catch：在最外层统一捕获所有异常
  2. 分层分类：每层有自己的错误处理策略，通过 ErrorClassifier 统一分类
  3. 让每个模块完全自治：各自处理自己的错误
- **决定**：选择**分层分类（选项 2）**。
- **理由**：集中式太粗糙（无法区分可恢复和不可恢复）；完全自治会导致错误处理逻辑重复且不一致。分层分类在每层处理自己能处理的错误，无法处理的统一上报。
- **后果**：需要定义清晰的错误分类标准（本文档 1.2 节）；ErrorClassifier 是一个轻量组件，增加的复杂度极小。

## 补充二：I/O Layer 与 Adapter 层详细设计（对应 Mission 1 扣分 -5）

### 2.1 I/O Layer 请求-响应生命周期

```
┌─────────────────────────────────────────────────────────────────────┐
│                    I/O Layer 请求-响应生命周期                        │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  ┌─────┐     ┌──────────┐     ┌──────────┐     ┌──────────────┐   │
│  │用户  │     │Transport │     │ I/O      │     │ Agent        │   │
│  │输入  │────►│ Handler  │────►│ Router   │────►│ Framework    │   │
│  │     │     │          │     │          │     │ Core          │   │
│  └─────┘     │· HTTP    │     │· 认证    │     │              │   │
│              │· CLI     │     │· 限流    │     │              │   │
│              │· IDE WS  │     │· 路由    │     │              │   │
│              └──────────┘     │· 请求    │     └──────────────┘   │
│                               │  规范化  │            │            │
│                               └──────────┘            │            │
│                                                      │            │
│  ═══════════════════ 处理中 ═══════════════════════════╪═══════════ │
│                                                      │            │
│                                                      ▼            │
│  ┌─────┐     ┌──────────┐     ┌──────────┐     ┌──────────────┐   │
│  │用户  │     │Transport │     │ I/O      │     │ Agent        │   │
│  │输出  │◄────│ Handler  │◄────│ Router   │◄────│ Framework    │   │
│  │     │     │          │     │          │     │ Core          │   │
│  └─────┘     │· 格式化  │     │· 错误    │     │              │   │
│              │· 流式    │     │  包装    │     │              │   │
│              │· 响应    │     │· 日志    │     │              │   │
│              └──────────┘     └──────────┘     └──────────────┘   │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

**各阶段职责**：

| 阶段 | 输入 | 输出 | 职责 |
|------|------|------|------|
| Transport Handler | 原始请求（HTTP/CLI/WS） | 标准化的 `IncomingRequest` | 协议解析、流式适配、连接管理 |
| I/O Router | `IncomingRequest` | `AgentRequest` | 认证、限流、请求规范化、路由到 Agent Framework |
| Agent Framework | `AgentRequest` | `AgentResponse` | 核心处理（已在 Mission 1 中定义） |
| I/O Router（响应） | `AgentResponse` | `OutgoingResponse` | 错误包装、日志记录、响应规范化 |
| Transport Handler | `OutgoingResponse` | 原始响应 | 格式化（JSON/流式/富文本）、协议适配 |

**统一数据模型**：

```python
@dataclass
class IncomingRequest:
    """所有 Transport Handler 产出的标准化请求。"""
    session_id: str | None
    user_input: str
    attachments: list[Attachment] | None = None  # 文件、图片等
    metadata: dict[str, str] = field(default_factory=dict)
    stream: bool = False  # 是否要求流式响应
    source: str = ""  # 来源标识（"cli", "http", "ide"）

@dataclass
class AgentResponse:
    """Agent Framework 返回的标准化响应。"""
    success: bool
    content: str
    subtask_results: dict | None = None
    error: UserFacingError | None = None
    token_usage: TokenUsage | None = None

@dataclass
class UserFacingError:
    """面向用户的错误信息。"""
    category: str  # "transient" | "tool_error" | "budget" | "system"
    message: str   # 用户可读的错误描述
    recoverable: bool
    suggestion: str | None = None  # 建议用户怎么做
    completed_work: str | None = None  # 已完成的部分结果
```

### 2.2 Adapter 注册与发现机制

```
┌─────────────────────────────────────────────────────────────────┐
│                   Adapter 注册与发现                              │
├─────────────────────────────────────────────────────────────────┤

│  │                               │                               │

│  │  "data_analysis"  → DataAdapter      ✅                      │

│  │                               │                               │

│  │  list_available() → [...]     │                               │
│  └──────────────┬───────────────┘                               │
│                 │                                                │
│                 ▼                                                │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │  resolve(task) 选择流程                                    │   │
│  │                                                           │   │
│  │  1. 遍历所有已注册 Adapter                                 │   │
│  │  2. 调用 adapter.can_handle(task) → score (0.0~1.0)      │   │
│  │  3. 选择 score 最高的 Adapter                              │   │

│  │                                                           │   │
│  │  通用 Adapter（built-in）：                                │   │
│  │  · 不注入任何专用系统提示                                   │   │
│  │  · 输出格式为纯文本                                        │   │
│  │  · 兜底保证任何任务都能被处理                               │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                  │
│  Adapter 失败处理：                                              │
│  ────────────────                                               │
│  · prepare_context() 抛异常 → 跳过该 Adapter，用通用 Adapter    │
│  · Adapter 注册后健康检查失败 → 标记为 unhealthy，不参与 resolve │
│  · 运行时发现 Adapter 质量差 → Reflection Engine 可反馈          │
│    给 Orchestrator，后者下次 resolve 时降低该 Adapter 的权重      │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**V1 简化**：Adapter 在启动时注册，运行时不变更。不搞动态热加载——那是 V2 的事。

### ADR-007: Adapter 选择用关键词匹配而非 LLM 分类

- **背景**：需要在多种 Adapter 中选择最合适的一个。两种思路：关键词匹配（简单快速）和 LLM 分类器（精准但有 Token 成本）。
- **选项**：
  1. 关键词匹配：维护关键词 → score 映射表
  2. LLM 分类器：调一次 LLM 判断任务类型
  3. 混合：关键词快速匹配，低置信度时回退到 LLM
- **决定**：V1 选择**关键词匹配（选项 1）**，V2 预留 LLM 回退。
- **理由**：关键词匹配零 Token 成本，且任务类型数量有限（<10），关键词表维护成本低。LLM 分类器的精度提升在 V1 阶段不值得额外的延迟和成本。
- **后果**：边缘场景可能选错 Adapter，但通用 Adapter 兜底不会出错，只是没有专用提示的优化效果。

### ADR-008: I/O Layer 采用统一数据模型而非直接透传

- **背景**：不同 Transport（HTTP/CLI/WS）的请求格式不同，需要标准化。
- **选项**：
  1. 直接透传：每个 Transport 直接调 Agent Framework，各自处理格式
  2. 统一模型：所有 Transport 先转为 `IncomingRequest`，再进入 Agent Framework
- **决定**：选择**统一模型（选项 2）**。
- **理由**：透传意味着认证、限流、日志等中间件在每个 Transport 中重复实现。统一模型让中间件只写一次，且添加新 Transport 时只需实现 Handler 层的转换。
- **后果**：增加了一层数据转换，但 `IncomingRequest` 结构简单，转换成本极低。

## 补充三：全局 Token 预算管理（对应 Mission 1 扣分 -3）

### 3.1 问题回顾

Mission 1 中，Session Manager 通过 `get_context_window(max_tokens)` 管理上下文 Token，但 Agent Core 的 LLM 调用（推理输出）和 Tool 调用（工具结果注入上下文）都消耗 Token，这些消耗没有与 Session Manager 的预算协调。可能导致：上下文已预留空间，但一次 LLM 推理就耗尽了剩余预算。

### ADR-009: Token 预算管理采用软分区而非硬隔离

- **背景**：Agent 系统中，上下文消息、LLM 输出、工具结果都在同一个 Token 池中消耗。需要一种分配机制防止某一方过度消耗。
- **选项**：
  1. 硬隔离：每个分区有固定上限，超出即拒绝
  2. 软分区：设定建议比例，允许弹性借用
  3. 先到先得：不做分区，谁先消耗谁先得
- **决定**：选择**软分区（选项 2）**。
- **理由**：硬隔离会导致浪费——比如简单任务上下文用不完，但输出被限制。先到先得太危险——上下文历史可能把输出空间挤占殆尽。软分区在大部分场景下效果接近硬隔离，但不会因僵化的比例导致资源浪费。
- **后果**：需要预留 10% 安全余量防止总超限；软分区的弹性借用规则需要在实现中仔细处理边界情况。

### ADR-010: 配置采用 4 级优先级合并而非单一配置源

- **背景**：Agent 系统需要大量可配置项（模型、超时、Token 预算等），且不同部署环境（开发/Docker/生产）的配置需求不同。
- **选项**：
  1. 全部通过命令行参数传入（如 `--model glm-5 --max-tokens 4096`）
  2. 全部通过单一配置文件
  3. 代码默认值 → 配置文件 → 环境变量 → 运行时参数（4 级优先级）
- **决定**：选择**4 级优先级（选项 3）**。
- **理由**：
  - 代码默认值保证零配置启动
  - 配置文件适合项目级设置（团队共享）
  - 环境变量适合部署环境差异（CI/CD、Docker）
  - 运行时参数适合单次会话级覆盖
- **后果**：合并逻辑需要正确处理覆盖和类型转换；配置冲突时需要清晰的错误提示。

### ADR-011: 配置采用结构化 YAML 而非 .env 文件

- **背景**：需要一种人类可读、支持嵌套结构的配置格式。
- **选项**：
  1. .env 文件：扁平键值对，Docker 友好但不支持嵌套
  2. JSON：结构化但可读性差（无注释、缩进严格）
  3. YAML：结构化 + 可读性好 + 支持注释
- **决定**：选择 **YAML**，环境变量作为补充。
- **理由**：配置项有嵌套结构（llm.temperature、session.token_budget），YAML 天然支持；注释方便团队理解配置意图；环境变量只用于敏感信息和部署覆盖。
- **后果**：需要引入 PyYAML 依赖（极轻量，无安全风险如果禁用 YAML 的任意对象解析）。

## Part B：对照 Mission 1 的反思

### B.1 这 4 个补充设计，是否让整体架构变得"过度设计"了？

**不，没有过度设计。** 但需要诚实地说，有两个补充设计的细节程度接近边界：

**合理的补充**（解决了真实的架构缺口）：
- **错误处理**（补充一）：这是任何生产系统都必须有的。Mission 1 中 Reflection Engine 只管"质量不好"的问题，但 API 超时、工具崩溃这些更基础的问题完全没覆盖。补充设计没有引入新的重量级模块——ErrorClassifier 是一个轻量分类器，CircuitBreaker 是成熟模式，复杂度可控。

- **配置管理**（补充四）：没有配置管理就意味着硬编码，硬编码意味着每次改参数都要改代码重新部署。4 级优先级是行业标准做法，YAML + 环境变量是常规选择，没有发明轮子。

**接近边界的补充**（V1 可以更简化）：
- **Token 预算管理**（补充三）：3 阶段降级策略设计得比较完整，但 V1 可能只需要最简单的实现——`if remaining < threshold: return partial_results`。详细的比例分配和紧急压缩可以 V2 再加。不过，Token 耗尽是 Agent 系统的真实痛点，提前设计好降级路径比出了问题再补要稳妥。

- **I/O Layer 详细设计**（补充四）：统一的 IncomingRequest / AgentResponse 模型是必要的（否则每个 Transport 重复实现），但 2 个 Adapter 的详细设计（CodeAdapter 和 DataAdapter）更像是实现指导而非架构设计。不过考虑到 Mission 1 的扣分原因是"I/O Layer 和 Adapter 层缺少实际交互流程"，补充这些详细设计是直接回应扣分项。

### B.3 最大的认知变化

**从"模块接口设计 = 架构完成"到"模块间的缝隙才是架构的真正难点"。**

Mission 1 我花大量精力定义了每个模块的接口——LLMProvider、ToolRegistry、SessionManager——觉得接口清楚了架构就清楚了。但自我评估时发现，模块之间的"缝隙"才是问题所在：

- **Token 预算**：Session Manager 和 Agent Core 各自管自己那一块 Token，中间没有协调机制。这是典型的"每个模块都对，但整体不对"的问题。
- **错误传播**：每个模块内部可以处理自己的错误，但跨模块的错误（工具失败 → Agent Core → Orchestrator → I/O Layer）没有统一的路径。
- **配置流动**：配置值从哪来、怎么到每个模块、能不能运行时改——这些问题在 Mission 1 中完全空白。

**教训**：架构设计的核心不是画模块框图和定义接口，而是设计模块之间的协议和协调机制。框图是静态的，协调是动态的。静态设计容易，动态设计难。

### B.4 实际价值评估

**对产品实际价值最大：错误处理与恢复策略（补充一）**

理由：Agent 系统在实际使用中，错误是常态而非例外——LLM API 会超时、工具会失败、Token 会耗尽。没有错误处理的 Agent 在遇到第一个 API 错误时就崩溃了，用户体验极差。补充一的设计让系统从"一碰就碎"变成"有弹性"，这是从 demo 走向产品的关键一步。

**最可能"写了但用不上"：CircuitBreaker 熔断器的 HALF_OPEN 探测机制**

理由：熔断器的 CLOSED → OPEN → HALF_OPEN 状态机是经典模式，在微服务架构中必不可少。但在 Deep Agent 的场景中，外部依赖主要是 LLM API（通常只有一个 Provider）和少量工具。如果 LLM Provider 挂了，HALF_OPEN 探测意义不大——用户更可能直接换一个 Provider 或等恢复。不过，CircuitBreaker 的核心价值（防止连续失败时无意义重试浪费 Token）是真实的，只是 HALF_OPEN 探测可能是过度实现。

**另一个"可能用不上"的细节：DataAdapter 的数据加载预检查（2.4 节）**

理由：这是一个锦上添花的优化——先读前 50 行了解数据结构再分析。实际中 LLM 通常能直接处理数据文件，不需要这个预检步骤。而且预检步骤增加了额外的 SubTask 和 Token 消耗。但如果数据文件很大（几百 MB），预检可以避免把整个文件灌入上下文的灾难——所以它在特定场景下有价值。
