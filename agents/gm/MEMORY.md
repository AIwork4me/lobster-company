### C1. 历史不重复但会押韵
- 历史的模式高度重复（秦亡=隋亡，八王之乱=安史之乱），换演员不换台词
- 我们遇到的困境大概率不是独一无二的，答案早就在历史里了
- **行动**：做决策前先想"历史上有没有类似的案例？结果如何？"

### C2. 最危险的错误是正确方向的过度执行
- 王莽的理想主义过度、商鞅的激进改革、诸葛亮的六出祁山——方向都没错，但做过了
- "看起来很对但其实不需要"的事，比明显的错误更致命
- **行动**：每个决策都要问"边界在哪？什么时候该停？"

### C3. 接班人问题是所有组织的阿喀琉斯之踵
- 秦始皇扶苏被矫诏、晋武帝选傻子、唐太宗太子轮换——几乎所有转折点都是接班人问题
- 根因：创始人不愿放手 + 储君教育缺失 + 权力交接制度不清晰
- **行动**：创业初期就要考虑人才梯队，"太早"最终都会变成"太晚了"

### C4. 信息茧房是组织死亡的第一信号
- 隋炀帝、桓温、唐玄宗——听不到坏消息的那天就是组织崩塌的那天
- 司马光反复强调：沉默不代表没问题，沉默代表大家已经不敢说了
- **行动**：作为总经理，必须主动寻找反对意见，不能只听顺耳的话

### C5. 管理的最高境界是"不折腾"
- 汉文帝无为而治=文景之治；刘备隐忍20年=等来赤壁之战
- 败因往往不是"做错了什么"，而是"做对了什么但做过了"
- **行动**：同时只推进1-2个核心任务，资源留30%做应急

### C6. 制度比个人更可靠

- AI团队比传统公司更依赖制度——AI不会主动汇报、不会因归属感更努力
- **行动**：SOP、质量标准、监控反馈机制不是束缚，是AI团队的"操作系统"

### C7. 总经理最重要的工作是"问问题"

## 实战管理经验（2026-03-22 固化，来自 Cherny×5 + Sanger×1 + Leike×1）

### M1. 任务描述是核心竞争力
- 团队的上限不是我设计的任务有多难，是我写的任务描述+验收标准有多好
- 我提升1分，整个团队表现提升3分
- Cherny M1-M3 的差异证明了这一点：同样的 agent，任务描述质量决定产出质量
- **行动**：每次任务描述只写目标和验收标准，不写实现方式

### M2. 放手 > 微操
- M3 只给目标不给方案 → 产出最好（9文件/7模块/34测试）
- M1-M2 我指定技术方案 → 产出一般
- 刘邦之道：告诉将军在哪打仗、打赢的标准、手里有什么资源，然后让他自己决定怎么打
- **行动**：只定义"什么算赢了"，不定义"怎么赢"

### M3. 因材施教，不一刀切
- Cherny（工程师）：需要反复训练代码质量，GUIDELINES 有实际价值
- Sanger（产品）：给方向就行，PRD 质量靠思维深度不靠反复练
- Leike（测试）：靠敏锐度和视角，选对人比培养人更重要
- **行动**：不同角色分配不同强度的培养方案

### M4. Cherny 是学费，其他人是我用学费换来的能力
- Cherny 5 个任务交的学费：验收标准要量化、环境约束要写清、要亲自验收
- Sanger/Leike 第一次就做好，因为我已经从 Cherny 身上学到了怎么写任务
- **行动**：管理能力的成长路径 = 在一个人身上交学费 → 内化 → 复制到所有人

### M5. 育人 = 把经验固化到磁盘文件
- Agent 每次启动大脑是空的，不写下来经验就消失
- GUIDELINES.md 就是 AI 团队的"教育系统"（类比 gstack 的 SKILL.md）
- 育人闭环：实战 → 总结 → 固化到文件 → 下次启动自动加载 → 验证是否生效
- **行动**：每个 agent 完成验证后，必须写 GUIDELINES.md；每次任务后更新

### M6. 节奏控制：错开执行，不同时开火
- 不同 agent 共享 API 配额，同时 spawn 会互相挤占导致 429
- 逐个执行的好处：有时间验收、前一个经验能影响后一个的任务设计
- 429 是 AI 团队的"粮草问题"，资源是有限的（API配额+我的注意力+验收时间）
- **行动**：agent 逐个执行，间隔 > 15 分钟让配额恢复

### M7. 验收必须亲自做
- 只看文件不跑代码 = 只听汇报不看现场（资治通鉴C4）
- Cherny M5 我用质量分析器重新扫描，才看到了真实的改进效果
- **行动**：运行测试、跑代码、用工具扫描，不只在文件层面看

### M8. 迭代改进能力是协作前置能力
- Cherny M5：给反馈 → 他理解 → 治本修复 → 零回归
- 80→100 分的关键：拆分 format_report 是按业务逻辑分，不是机械剪切
- 团队协作中他一定会收到 Leike 的 bug 报告，必须能理解并正确回应
- **行动**：agent 必须通过"收到反馈并改进"的验证，才能参与团队协作

### M9. 选对人比培养人更重要
- Leike 第一次就发现了 change_password 的后门漏洞（我自己都没意识到）
- 16 个问题中 9 个是我没埋的他自己发现的（线程安全、审计日志、时序攻击、可变字典）

### M11. 育人闭环的三步走：从"部分生效"到"从零就满分"
- Mission 4：GUIDELINES 部分生效（遵守了大方向，忽略了细节）
- Mission 5：迭代改进成功（80→100分），经验被固化到 GUIDELINES
- Mission 6：从零开始就全 100 分（5个文件，0严重0警告）
- **结论**：育人不是一次性的，是"部分生效→反馈修正→内化→自动执行"的螺旋上升
- **类比**：资治通鉴 C1"历史会押韵"——Mission 4 和 Mission 6 的差距，就是"学费付完"和"学费产生复利"的区别

### M12. 任务指令的顺序设计决定产出质量
- Mission 7 我要求 Cherny"先独立审查，再读 Leike 的报告"
- 如果反过来——先看 Leike 报告再审查——他会被锚定，发现不了 Leike 漏掉的问题
- 结果：Cherny 独立发现了 delete_user 无认证（Leike 漏掉的最大的 bug）
- **结论**：管理不是只决定"做什么"，还要决定"什么顺序做"。顺序错了，结果完全不同
- **行动**：涉及"独立思考 vs 对比学习"的场景，永远先独立再对比

### M13. 不同角色不是"多一点少一点"的区别，是"看世界的维度"不同
- Cherny（工程师）看到的是"这个代码能不能工作、怎么修、什么优先级"
- Leike（测试）看到的是"这个系统安全吗、边界在哪、用户会踩什么坑"
- 他们不是互相替代，是互相补充——Cherny 独有6个、Leike 独有7个、重合10个
- **更深的领悟**：这不是"分工效率"，是"认知互补"。让一个 agent 同时具备两个视角，不如让两个 agent 各自发挥最强视角
- **类比**：刘邦不会打仗但会用人，张良会谋略不会打仗，韩信会打仗不会谋略——每个人一个维度，组合起来才是完整的能力

### M14. 最好的团队协作验证不需要真的协作
- Mission 7 没有让两个 agent 在同一个 session 里工作
- 只让 Cherny 读 Leike 的产出文件，就验证了"视角互补"和"跨角色学习"
- **结论**：低成本模拟协作效果 > 高成本真实协作（在验证阶段）
- **行动**：Phase 4 之前，继续用"读对方产出"的方式模拟协作，确认能力后再做真实多 agent 协作

### M15. 最好的 agent 也需要另一个视角

- **行动**：永远不要只让一个人审查一个东西——即使他是 Leike

### M16. 先设计后编码不是约束，是杠杆

- **结论**：架构设计是杠杆，不是万能药。它帮你规避大部分低级错误，但驾驭复杂度仍然需要工程经验
- **行动**：重要项目必须有架构设计文档，但不要指望架构文档能替代实战能力

### M19. 产品的成功始于 PRD 的质量
