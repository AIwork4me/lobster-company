## 六、API Key 独立配额实验（17:00-18:00）

### 问题
董事长发现 Sanger 和 Leike 的 API Key 在智谱后台显示"未使用"。所有 subagent 实际都在消耗 GM 的配额。

### 根因
- 我把 Key 存在了每个 agent 的 .env 文件里，假设"配置即生效"
- 实际上 OpenClaw 的 sessions_spawn **不会读取 .env 文件**
- 真正的 Key 路由在 `~/.openclaw-autoclaw/openclaw.json` 的 `models.providers` 里
- 所有 spawn 都用 GM 的 provider，所以 GM 的 Key 承担了全部消耗

### 解决方案（已验证）
在 openclaw.json 里为每个 agent 注册独立 provider，spawn 时通过 `model` 参数指定：
```
zai__sanger/glm-5-turbo → 用 Sanger 的 Key
zai__cherny/glm-5-turbo → 用 Cherny 的 Key
zai__leike/glm-5-turbo → 用 Leike 的 Key
zai__steinberger/glm-5-turbo → 用 Steinberger 的 Key
```

### 实验结果
- ✅ 实验1：注册 Sanger provider → spawn 最小任务 → 智谱后台确认 Sanger Key 已使用
- ✅ 实验2：批量注册 Cherny/Leike/Steinberger → JSON 格式验证通过
- 实验结论：方案成立，彻底解决了 API 配额共享问题

### 经验教训（M26）
- **"配置即生效"是假设，不是事实** — 写了文件不等于系统会读
- **务实的方法：先查文档、再设计实验、用实践验证** — 不要假设，要证明
- **429 频繁的真正原因找到了** — 不是智谱限速太严，是所有 agent 挤同一个 Key

### 第五次（API Key 独立配额实验 + Steinberger 首战）

- **写文件 ≠ 系统会读** — .env 文件写了 Key 但 OpenClaw 从不读取它，所有 subagent 都在用 GM 的配额。这不是"小疏忽"，是"把假设当事实"的典型案例
- **问题表象 vs 问题根因** — 429 频繁，表面看是"智谱限速太严"，根因是"所有 agent 挤同一个 Key"。解决表象（等 15 分钟再 spawn）只能缓解，解决根因（独立 provider）才能根治
- **最小实验验证假说** — 不需要搞复杂的系统改造，一个最小 spawn（"回复一句话"）就能验证整个路由机制是否生效
- **基础设施问题会伪装成业务问题** — 429 让我以为"需要控制并发节奏"，实际上需要的是"解决资源隔离"
- **Steinberger 的架构师视角是 Cherny 缺失的那块** — Cherny 看到"怎么编排"，Steinberger 看到"确定性 vs 非确定性的本质区别"。这不是能力高低，是看问题的维度不同
- **P0 bug 是跨角色评审的价值** — Cherny 和 Leike 都没发现的 bug（Stage.execute 返回值检查），Steinberger 第一次就看出来了。互补性再次验证

### 第六次（Steinberger Mission 2 迭代改进后）

- **迭代改进的验收方式因角色而异**：Cherny 的迭代是"代码分数 80→100"，Steinberger 的迭代是"架构分数 82→90"。不同角色的"进步"长不同的样子，验收标准也要因人而异
- **"缝隙"比"模块"更重要**：Steinberger 的认知变化——架构的核心不是画模块框图，而是设计模块间的协议和协调机制。框图是静态的，协调是动态的
- **90 分的架构师比 100 分的更有价值**：Steinberger 给自己 90 而不是 95+，因为他还看到了安全、可观测性、持久化三个缺口。能看见自己看不见的东西，才是顶级能力
- **增量补充 > 推翻重来**：Mission 2 不是重写架构，是对 4 个精准缺口的填补。管理启示：纠正团队问题时，精准补缺比全面改革更有效
- **好的架构师能预判"什么会没用"**：Steinberger 明确说 HALF_OPEN 熔断探测和 DataAdapter 预检查"可能用不上"。这不是不自信，是对场景的深度理解——知道什么时候该停，比知道什么时候该做更难
