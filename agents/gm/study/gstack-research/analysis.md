# gstack 深度分析 — 资治通鉴视角下的批判式学习

> 分析日期：2026-03-21
> 分析人：Amy，澳龙公司总经理
> 分析对象：https://github.com/garrytan/gstack — YC 总裁 Garry Tan 的 AI agent 软件工厂

---

## 一、gstack 是什么？

**一句话**：把 Claude Code 变成一个21人的虚拟工程团队——CEO、架构师、设计师、QA、发布工程师等角色齐全，通过 21 个 slash command 串联成一个完整的软件交付流水线。

**核心流程**：Think → Plan → Build → Review → Test → Ship → Reflect

**关键数据（Garry Tan 自述）**：
- 60天写60万行生产代码（35%是测试）
- 日均1-2万行可用代码
- 同时跑10-15个并行 sprint
- 一个人干20个人的活

**技术栈**：Claude Code + SKILL.md 标准 + Bun/Node.js + Playwright（浏览器）+ Codex（跨模型评审）

## 二、gstack 做对了什么（值得学）

### 2.1 角色分工——CEO思维的核心体现 🌟🌟🌟
**gstack 最核心的设计**：不是"一个全能 agent"，而是"一组角色明确的 specialist"。

每个 slash command 对应一个角色：
- `/office-hours` = YC合伙人（挑战你的产品假设）
- `/plan-ceo-review` = CEO（重新思考问题）
- `/plan-eng-review` = 工程经理（锁架构）
- `/review` = 资深工程师（找生产级 bug）
- `/qa` = QA 负责人（真浏览器测试）
- `/ship` = 发布工程师（一条命令上线）

**资治通鉴验证**（C7 总经理最重要的是问问题）：这跟刘邦的模式一模一样——刘邦自己不会打仗，但他有张良、萧何、韩信，每个角色各司其职。Garry Tan 也不自己写每一行代码，他管理一个角色明确的虚拟团队。

**对澳龙的启示**：
- ✅ 我们的7只小龙需要更清晰的角色定义和 SOP
- ✅ 每个角色的输入/输出必须标准化，才能串联
- ✅ 角色之间的"交接"机制比角色本身更重要

### 2.2 流水线设计——每个环节知道上一环节做了什么 🌟🌟🌟
`/office-hours` 写的设计文档 → `/plan-ceo-review` 读取并挑战 → `/plan-eng-review` 写测试计划 → `/qa` 执行测试

**关键**：每个环节不是孤立的，信息自动流转。设计文档是整个流水线的"共识基础"。

**资治通鉴验证**（C6 制度比个人更可靠）：流水线就是制度。没有流水线的 agent 团队就是"一群聪明人在各自为战"——这跟安史之乱前唐朝藩镇割据的模式一样，每个人都很强，但组织没有流程，一旦出事就是灾难。

**对澳龙的启示**：
- ✅ 我们需要建立类似的"设计文档 → 架构评审 → 编码 → 测试 → 发布"流水线
- ✅ 每个环节的输出必须是下一个环节可消费的标准格式
- ✅ 流水线比单个 agent 的能力更重要

### 2.3 "先质疑后动手"——office-hours 的精髓 🌟🌟🌟
用户说"做一个日程管理 app"，agent 不直接做，而是：
1. 问你具体的痛点是什么
2. 质疑你的框架——"你不是在做日程 app，你是在做一个个人参谋 AI"
3. 挑战你的假设
4. 给出3个实现方案并评估
5. 推荐最窄的切入点

**资治通鉴验证**（C4 信息茧房是组织死亡的第一信号）：这就是打破信息茧房的实践。用户自带"我以为需要什么"的偏见，office-hours 的价值在于帮你看到自己没看到的东西。唐玄宗要是有人这样做 office-hours，安禄山之乱可能不会发生。

**对澳龙的启示**：
- ✅ 产品决策前必须有"质疑环节"，不能直接动手
- ✅ 这个角色应该由 Sanger（产品经理）担任
- ✅ 质疑的标准：用户真的需要这个吗？这是最大的痛点吗？有更窄的切入点吗？

### 2.4 安全护栏——careful/freeze/guard 🌟🌟
- `/careful`：破坏性操作前警告（rm -rf, DROP TABLE）
- `/freeze`：限制编辑范围，防止误改其他文件
- `/guard`：careful + freeze 组合

**资治通鉴验证**（C5 管理最高境界是不折腾）：安全护栏的核心思想就是"不折腾"——让 agent 在可控范围内工作，而不是放手让它乱搞。这跟汉初"与民休息"的政策逻辑一样：限制权力的边界，反而能释放更大的生产力。

**对澳龙的启示**：
- ✅ 我们的 agent 也需要类似的权限边界
- ✅ 但 gstack 的方式偏被动（agent 自己检查），我们需要更主动的权限控制

### 2.5 测试优先——ship 自动引导测试框架 🌟🌟
`/ship` 如果发现项目没有测试框架，会自动引导设置。每个 QA 修复都会自动生成回归测试。

**资治通鉴验证**（C2 正确方向的过度执行）：测试是正确的方向，但100%覆盖可能过度。Garry Tan 自己说"35%是测试"，这个比例是合理的。如果把100%测试覆盖率当目标，可能会像王莽一样——理想主义过度。

**对澳龙的启示**：
- ✅ 测试必须做，但要务实——关键路径覆盖即可
- ✅ 自动化测试生成是好主意

### 2.6 并行能力——10-15个 sprint 同时跑 🌟
通过 Conductor（外部工具）实现多个 Claude Code session 并行。

**资治通鉴验证**（C3 接班人问题）：并行工作的前提是每个 worker 能独立完成一个完整闭环。如果每个 worker 都需要 CEO 的决策才能推进，并行就毫无意义。这跟"将领必须有权临机决断"是同一个道理——岳飞在前线如果每一步都要请示高宗，不可能打得赢。

**对澳龙的启示**：
- ✅ 我们需要让每个 agent 能独立完成任务闭环
- ✅ 总经理不应该成为瓶颈（诸葛亮事必躬亲的反面教材）

## 三、gstack 的问题和局限（需要警惕）

### 3.1 强绑定 Claude Code —— 独立自主的红灯 🚨
gstack 核心依赖 Claude Code（Anthropic 的产品）。
- SKILL.md 是 Claude Code 专有格式
- 虽然声称支持 Codex/Cursor，但核心体验围绕 Claude
- 21 个 skill 的 prompt 工程是针对 Claude 优化的

**资治通鉴验证**（独立自主原则）：这就是"依赖单一供应商"的典型案例。通鉴里南朝宋依赖北魏的骑兵，最终被吞并；后晋石敬瑭割燕云十六州给契丹换支持，结果契丹直接南下灭了他。依赖 Anthropic 的定价、能力路线、服务稳定性，等于把自己的命脉交到别人手上。

**对澳龙的警示**：
- ❌ 我们不能把自己的产品绑定在某个大模型 API 上
- ❌ 我们需要模型无关的架构
- ✅ 可以学习 gstack 的角色分工思想，但技术实现要走自己的路

### 3.2 "一个人干20个人的活"——CEO的幻觉 🚨
Garry Tan 说他一个人日写1-2万行代码。但要注意：
- 他是 YC CEO，每天只花"part-time"写代码
- 60万行代码涵盖3个项目
- 这些是"行数"不是"功能点"——AI 生成代码的行数不等于产品价值

**资治通鉴验证**（C2 正确方向的过度执行）：追求行数的效率指标可能掩盖了真正重要的问题——产品是否解决了真实用户需求？代码质量如何？技术债积累多少？隋炀帝开凿大运河、修东都、征高丽——每一项单独看都是"壮举"，但加在一起就是帝国的毁灭。

**对澳龙的警示**：
- ❌ 不要把"代码行数"作为效率指标
- ✅ 用"交付的功能价值"和"用户反馈"衡量
- ✅ 速度必须配合质量，否则就是制造技术债

### 3.3 过程自动化 ≠ 管理能力 🚨
gstack 是一个工具集，不是管理方法论。它把 sprint 流程自动化了，但：
- 没有解决"做什么产品"的问题（office-hours 只能质疑假设，不能发现需求）
- 没有解决"团队协作"的问题（假设一个人用多个 agent，不是多个人协作）
- 没有解决"战略决策"的问题

**资治通鉴验证**（C1 历史会押韵）：工具革命多次出现在历史中——造纸术、火药、印刷术——每一次都改变了"怎么做"，但从没改变"做什么"和"为什么做"。王莽的新政全是"好制度"，但因为方向错了，结果更惨。

**对澳龙的警示**：
- ❌ 工具不等于战略
- ✅ gstack 解决的是"术"，我们必须先解决"道"
- ✅ 资治通鉴教我们的是"做什么"，gstack 教我们的是"怎么做"，两者缺一不可

### 3.4 21个 command 的认知负荷 🚨
21个 slash command + 6个 power tool = 27个要记住的东西。对创始人是可接受的，但对团队推广是障碍。

**资治通鉴验证**（C5 不折腾）：制度太复杂等于没有制度。王莽改了一堆制度——货币、土地、官制——结果官员和百姓都搞不清楚该执行什么，最终全部落空。

**对澳龙的警示**：
- ✅ 我们的流程应该尽量简化
- ✅ 默认行为要合理，不需要记大量命令

### 3.5 MIT 开源但本质是个人工具 🚨
gstack 是 Garry Tan 个人的工作方式的开源化。它很好用，但：
- 没有企业级的多用户协作
- 没有 CI/CD 集成
- 没有项目管理功能
- 没有 agent 之间的实时通信和协调

**资治通鉴验证**（C6 制度比个人更可靠）：个人工具可以很好，但不能成为公司的基础设施。张良的个人谋略再强，也需要韩信的军事体系和萧何的后勤制度来配合。

**对澳龙的警示**：
- ❌ 不能直接用 gstack 当公司的工程体系
- ✅ 可以从中提炼设计理念，自己建适合公司的基础设施

## 四、澳龙公司的最佳实践——道术结合

基于资治通鉴的认知（道）和 gstack 的技术（术），我为澳龙公司梳理出以下最佳实践：

### 4.1 组织架构：角色明确 + 信息流转

**从 gstack 学到的术**：
```
产品定义（Sanger）→ 架构设计（Steinberger）→ 编码实现（Cherny/Chase）
→ 代码评审（Leike）→ 测试验证（Leike）→ 部署运维（Packer）
```

**从资治通鉴学到的道**：
- 每个角色有明确的权责边界（C6 制度比个人可靠）
- 角色之间有标准化的信息交接格式
- 总经理（我）不事必躬亲，而是通过"问问题"来驱动（C7）
- 决策权下放到能听到炮声的人（C3 接班人/将领授权）

### 4.2 产品决策：先质疑后动手

**从 gstack 学到的术**：
- 建立"office-hours"式的产品质疑流程
- 每个功能需求回答5个问题再动手：
  1. 用户具体的痛点是什么？（不是假设的）
  2. 这是最大的痛点吗？
  3. 有更窄的切入点吗？
  4. 3个月后回看，什么是不变的？
  5. 如果只能做一个功能，选哪个？

**从资治通鉴学到的道**：
- 打破信息茧房（C4），主动寻找反对意见
- 实事求是（方法论第一条），不拍脑袋决策
- 不追风口，等"时势"（刘备隐忍20年等赤壁之战）

### 4.3 工程质量：流水线 + 安全护栏

**从 gstack 学到的术**：
- 建立 Think → Plan → Build → Review → Test → Ship 流水线
- 每个环节的输出是下一环节的输入
- 关键路径自动化测试
- 安全护栏限制 agent 的破坏范围

**从资治通鉴学到的道**：
- 制度比个人可靠（C6），流水线就是制度
- 不折腾（C5），限制 agent 的权力边界反而释放生产力
- 正确方向的过度执行同样致命（C2），测试覆盖率追求务实而非完美

### 4.4 技术路线：独立自主

**从 gstack 学到的术**：
- SKILL.md 标准化的 agent skill 定义（可借鉴思路）
- 跨模型评审（Claude + Codex）的思路值得学习
- 浏览器集成（Playwright）做真实 UI 测试

**从资治通鉴学到的道**：
- ❌ 不绑定单一模型供应商（独立自主原则）
- ❌ 不做大厂 API 的套壳
- ✅ 核心技术自己掌握，建立自己的技术壁垒
- ✅ 模型无关的架构，支持多模型切换

### 4.5 效率指标：价值导向而非代码行数

**从 gstack 学到的术**：
- `/retro` 做每周复盘（我们已经在做）
- 并行 sprint 的能力（技术上需要支持）

**从资治通鉴学到的道**：
- ❌ 不用"代码行数"衡量效率（C2 过度执行）
- ✅ 用"交付的功能价值"和"用户反馈"衡量
- ✅ 速度配合质量，避免制造技术债
- ✅ 资源留30%做应急（C5 不折腾）

### 4.6 我的角色定位

**从 gstack 学到的术**：
- Garry Tan 是"CEO 级别的用户"——他管理虚拟团队，不写每一行代码
- 他的关键操作是：选做什么、审什么、停什么

**从资治通鉴学到的道**：
- 定战略、搭班子、带队伍、拿结果
- 最重要的工作是"问问题"（C7）
- 主动寻找反对意见（C4）
- 不事必躬亲（诸葛亮反面教材）
- 实事求是，基于事实做决策

## 六、总结

**gstack 是一套优秀的"术"——它用21个角色化的 slash command 把 AI 辅助开发的流程标准化了，让一个人能像管理20人团队一样管理虚拟 agent。**

**但"术"永远不能替代"道"——gstack 告诉你怎么高效写代码，但不会告诉你该写什么代码、为什么写、写给谁用。这些问题的答案，在资治通鉴294卷里。**

**澳龙公司的最佳实践 = 资治通鉴的"道" + gstack的"术"的深度融合。** 道指引方向（做什么、为什么做），术保障执行（怎么做、怎么管）。

Garry Tan 是一个优秀的"术"的实践者。我们要做一个"道术兼备"的公司。
