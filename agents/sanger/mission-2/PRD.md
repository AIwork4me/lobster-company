# PRD: PR Queue（Code Review 流程管理工具）

## 产品定位

给 GitHub/GitLab 的 Pull Request 加上催办机制和审查质量检查，让代码审查不再"没人看、看了白看"。

## 目标用户

5-30 人技术团队的 Tech Lead 或工程经理。他们每周花 2-3 小时催人 Review，因 Review 不充分导致的线上 bug 每月至少 1-2 次。

## 核心场景

**场景 1：PR 无人理睬**
周一下午，后端工程师小明提交了一个涉及支付逻辑的 PR。到了周四，这个 PR 还是 Open 状态，零条评论。PR Queue 自动在 Slack 频道发提醒，同时给指定 Reviewer 发通知："PR #342 等待 Review 已超过 24h，变更涉及 `payment` 模块"。Reviewer 当天完成了审查。

**场景 2：走过场式 Review**
周五下午赶发布，Reviewer 在 PR 里只留了一句 "LGTM"，3 行代码变更了数据库 schema。PR Queue 检测到 LGTM 评论 + 无行内批注 + 涉及高风险目录，自动标记为"需要二次审查"，并通知 Tech Lead 确认。

**场景 3：Review 积压看板**
Tech Lead 每天打开 PR Queue Dashboard，看到一张表：当前 Open 的 PR 共 12 个，其中 4 个等待超过 48h，2 个标记为高风险。点击进入可直接跳转对应 PR 页面，一键催办。

## MVP 功能清单（按优先级排序）

1. **P0 PR 超时自动催办** — 监听仓库 PR 事件，PR 等待超过可配置的时间阈值（默认 24h）时，通过 Slack/飞书 Webhook 发送通知给指定 Reviewer，通知内容包含 PR 标题、作者、等待时长、变更文件列表。
2. **P0 审查质量标记** — 检测"LGTM""看起来没问题""👍"等无实质内容的 Approval，结合变更文件路径和代码行数（如单次变更 >200 行、涉及 `migration`/`schema`/`payment` 等关键词目录），自动在 PR 上打标签（如 `needs-rereview`）并通知 Tech Lead。
3. **P1 Review 积压看板** — 一个只读 Web 页面，列出当前仓库所有 Open PR，按等待时长排序，显示：PR 标题、作者、等待时间、Reviewer 列表、风险等级标签。支持按仓库、时间范围筛选。
4. **P1 审查人轮值分配** — 管理员配置代码目录与 Reviewer 的映射规则（如 `src/payment/**` → @alice, @bob），新 PR 创建时自动按规则建议 Reviewer，并在 PR 评论中 @ 提及。
5. **P2 每周审查报告** — 每周一自动生成上周 Review 数据摘要：平均 Review 时长、LGTM 标记次数、超时 PR 数量，发送到指定 Slack/飞书频道。

## 不做什么

- **不做自动代码审查（AI 分析代码逻辑）**：MVP 阶段工具定位是流程管理，不是代码质量分析。自动代码审查是独立产品方向，技术复杂度完全不同。
- **不做 PR 内联评论编辑或代码建议**：这侵入了开发者工具链的核心交互区，容易引发抵触，且需要深度 IDE 集成，ROI 不高。
- **不做多平台 UI 界面（移动端、桌面客户端）**：目标用户是工程师，Web Dashboard 足够。渠道通知通过已有 IM（Slack/飞书）触达，不需要额外客户端。
- **不做 Jira/Linear 工单联动**：工具聚焦 PR 流程本身，与项目管理工具的联动是后续扩展，MVP 不引入额外依赖。

## 成功指标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| PR 平均等待审查时长 | 从 72h 降至 24h | 核心价值指标 |
| 无实质评论的 Approval 比例 | 下降 50% | 衡量审查质量改善 |
| 用户（Tech Lead）周活跃度 | ≥3 天/周 | 衡量产品粘性 |
| 单仓库接入时间 | ≤15 分钟 | 衡量上手门槛 |
