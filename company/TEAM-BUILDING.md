# 🦞 澳龙公司 — 建队方案 v2（TDD 模式）

> 作者：太宗（CEO） | 2026-03-18
> 方法论：实事求是 + TDD（测试驱动）——小步快跑，一边创建一边验证

---

## 一、核心策略转变

| 旧思路（v1） | 新思路（v2 TDD模式） |
|-------------|---------------------|
| 先建好全部队伍，再全面出击 | 创建一只，验证一只，能打了再建下一只 |
| 先侦察、再架构、再产品、再开发 | 每只小龙虾创建后**立即给真实任务**，验证通过才算"建好" |
| 串行推进，等情报回来再动 | 并行创建+验证，不等不靠 |
| 追求"完整的队伍" | 追求"每只都能打" |

**TDD 建队三原则：**
1. **Red** — 创建小龙虾，给它一个它应该能完成的任务
2. **Green** — 它完成了，验证合格，这只算"建好"
3. **Refactor** — 根据实战表现优化它的 SOUL.md 和 AGENTS.md，让它更强

**不合格怎么办？** 不藏不掩，分析失败原因，改了再试。改到能打为止。

## 二、TDD 建队流水线

### 阶段1：验证侦察兵（1只）—— 值得打吗？

**Red：** 激活 **Richards（侦察兵）**，下达真实任务：
> "调研 Deep Agent 产品市场，找到最痛的用户群体和最明确的痛点，3天内交付。"

**Green：** 验收标准——
- ✅ 有具体的目标用户画像（不是"开发者"这种泛泛的词）
- ✅ 有真实的痛点描述（有数据、有来源）
- ✅ 有竞品格局判断（谁强谁弱、机会在哪）
- ✅ 结论清晰可决策（"我们应该做X方向"）

**验证结果：**
- 通过 → 进入阶段2
- 不通过 → 分析原因，优化 Richards 的能力定义，重新派任务
- 完全没方向 → 说明"做 Deep Agent 产品"这个大方向需要重新审视（这是最该提前知道的事）

**为什么要先验证侦察兵？** 因为如果侦察回来的结论是"这仗不值得打"，后面建再多队伍都是浪费。**侦察兵是整个建队的验证点。**

### 阶段2：验证产品经理（+1只）—— 做什么？

**前提：** Richards 侦察结果已拿到且合格

**Red：** 激活 **Sanger（产品经理）**，下达任务：
> "基于 Richards 的侦察报告，定义第一个产品方案——做什么、不做什么、MVP 包含哪些功能、成功的标准是什么。"

**Green：** 验收标准——
- ✅ 有清晰的产品定义（一句话能说清楚）
- ✅ 有明确的功能清单（MVP 只做3-5个核心功能）
- ✅ 有"不做什么"的清单（至少5条）
- ✅ 成功标准可量化（不是"用户喜欢"而是"N个用户完成Y任务"）

**同时：** Sanger 的方案交给 Richards 复核——"从市场角度看，这个方向对不对？"（交叉验证）

### 阶段3：验证架构师+AI工程师（+2只）—— 怎么做？

**前提：** Sanger 产品方案已拿到且合格

**Red：** 同时激活 **Steinberger（架构师）** 和 **Cherny（AI工程师）**，下达任务：
> - Steinberger："基于产品方案，设计整体技术架构。"
> - Cherny："Deep Agent 核心引擎的 PoC（概念验证）——让 Agent 完成一个真实的复杂任务。"

**Green：** 验收标准——
- Steinberger：架构图 + 技术选型理由 + 关键风险
- Cherny：**能跑通的 demo**——不是 PPT，是一个能实际运行的 Agent 完成一个真实任务

**关键验证点：** Cherny 的 PoC 是整个建队最核心的验证——如果 Deep Agent 的核心引擎做不出来，其他都是空的。**这等于是我们的技术可行性验证。**

### 阶段4：验证全栈+运维（+2只）—— 能不能交付？

**前提：** 架构确定 + AI 引擎 PoC 通过

**Red：** 激活 **Chase（全栈工程师）** 和 **Packer（运维工程师）**，下达任务：
> - Chase："基于架构，把 Cherny 的 PoC 包装成可用的产品原型。"
> - Packer："搭建部署环境，让产品能在服务器上跑起来。"

**Green：** 验收标准——
- Chase：用户能打开、能交互、能完成一个完整任务流程
- Packer：产品部署成功，外部可访问，稳定性不崩溃

**此时累计 6 只小龙虾已验证通过。**

### 阶段5：验证测试（+1只）—— 靠不靠谱？

**Red：** 激活 **Leike（测试工程师）**，下达任务：
> "对当前产品进行全面测试——功能、性能、安全、边界情况。"

**Green：** 验收标准——
- 发现的问题有优先级分类
- 每个问题有复现步骤
- 有明确的通过/不通过判定

**里程碑：** 到这一步，7只小龙虾全部验证通过，MVP 基本就绪。可以拿到真实用户面前了。

### 阶段6：扩展（按需）

| 触发条件 | 激活 |
|---------|------|
| 产品方向需要深度分析 | **Moura（战略分析师）** |
| 产品准备推向市场 | **Wu（增长运营）** |

## 四、验证不通过的处理

**绝不藏着掩着，快速暴露问题才是 TDD 的精髓。**

| 情况 | 处理 |
|------|------|
| 小龙虾任务不达标 | 分析原因 → 优化 SOUL.md/AGENTS.md → 重新派任务（最多重试2次） |
| 重试2次仍不达标 | 说明这个角色定义有问题，回到根本重新设计 |
| 某个阶段验证失败 | 暂停后续阶段，集中解决当前问题 |
| 侦察结论是"不值得打" | 立即汇报董事长，重新评估方向（这是好事，不是坏事） |

**核心原则：不通过比"假装通过"有价值一万倍。越早发现问题，成本越低。**

---

## 五、纪律

- ✅ 创建一只，验证一只，能打了再建下一只
- ✅ 每个阶段有明确的验收标准，不含糊
- ✅ 验证不通过就停下来解决问题，不跳过
- ✅ 每完成一个阶段，向董事长汇报进展
- ❌ 不同时激活所有小龙虾（那是摊大饼，不是建队伍）
- ❌ 不在没有验证通过的情况下进入下一阶段
- ❌ 不自己替小龙虾干活
