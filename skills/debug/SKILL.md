# 龙虾 Debug — Packer 级根因分析

**"没有调查就没有发言权" — 毛泽东**

---

## 铁律

> **不调查就修复 = 犯罪。** 每个修复必须有根因分析支撑。

---

## Step 1: 复现

1. 获取确切的错误描述
2. 确认复现条件：
   - 什么环境？（OS / Node版本 / 浏览器）
   - 什么操作？
   - 什么输入？
   - 频率？（必现 / 偶现 / 特定条件）
3. **如果在沙盒环境能复现 → 在沙盒复现**
4. **如果不能复现 → 收集更多现场数据**

```bash
# 收集环境信息
node --version; npm --version; git log --oneline -5
# 查看错误日志
# 根据项目结构找日志文件
```

---

## Step 2: 缩小范围

1. **二分法定位** — 用 git bisect 或手动排除法
2. **时间线** — 这个Bug什么时候第一次出现的？
3. **变更关联** — 那个时间点改了什么？

```bash
git log --oneline --since="2026-01-01" -- [相关文件路径]
git diff <good-commit>..<bad-commit> -- [相关文件路径]
```

---

## Step 3: 根因分析

问五个为什么（5 Whys）：

```
现象: [错误描述]
为什么1: [原因] → 因为...
为什么2: [原因] → 因为...
为什么3: [原因] → 因为...
为什么4: [原因] → 因为...
为什么5: [根因]
```

根因必须是**可操作的** — 不是"代码有Bug"，而是"第47行缺少空值检查"。

---

## Step 4: 修复方案

**三个层次：**

1. **热修** — 止血（立即）："在调用前加 null check"
2. **根修** — 消除根因（当天）："重构函数签名，不允许传入 null"
3. **防护** — 防止复发（本周）："加 TypeScript strict mode + 单元测试"

修复后必须：
- 确认原Bug不再复现
- 确认修复没有引入新问题
- 写回归测试

---

## Step 5: 事后复盘

```
## 根因分析报告

**Bug**: [描述]
**严重度**: 🔴/🟡/🟢
**影响范围**: [受影响的用户/功能]
**根因**: [5 Whys 结论]
**修复**: [热修 + 根修 + 防护]
**回归测试**: [测试描述]
**预防措施**: [下次如何避免同类问题]
```
