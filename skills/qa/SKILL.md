# 龙虾 QA — Leike 级测试工程

**"不验证的知识是假设，不是真知。" — 实践论**

---

## 哲学

QA 不是"找Bug"。QA 是**验证产品是否满足用户的真实需求**。

Leike 的三个原则：
1. **测试用户旅程，不测函数** — 用户不关心你的函数返回值，关心"我点了这个按钮发生了什么"
2. **每发现一个Bug，问"为什么测试没拦住"** — 不是"谁写的Bug"，是"系统哪里漏了"
3. **安全思维** — 不是"正常路径能用吗"，是"恶意/极端输入会怎样"

---

## Step 1: 理解要测什么

1. 读取变更范围：`git diff <base>...HEAD --stat`
2. 识别受影响的用户旅程
3. 列出需要测试的场景

**测试场景矩阵：**

| 场景 | 正常路径 | 边界条件 | 错误路径 | 极端情况 |
|------|---------|---------|---------|---------|
| 场景1 | | | | |
| 场景2 | | | | |

---

## Step 2: 执行测试

### 2.1 自动化测试（如果存在）

```bash
# 运行项目测试套件
# 根据项目配置选择命令
```

### 2.2 手动测试场景

对每个场景，按以下步骤：
1. **前置条件** — 设置测试环境
2. **操作步骤** — 具体的用户操作
3. **预期结果** — 应该发生什么
4. **实际结果** — 实际发生了什么
5. **截图/日志** — 证据

### 2.3 安全测试

- 输入验证：空值、超长字符串、特殊字符、SQL注入
- 认证授权：未登录访问、越权操作
- 数据泄露：API 返回多余字段、日志打印敏感信息

---

## Step 3: Bug 报告

每个 Bug 用统一格式：

```
### Bug #[N]: [标题]
- **严重度**: 🔴 Critical / 🟡 High / 🟠 Medium / 🟢 Low
- **场景**: [哪个用户旅程]
- **复现步骤**:
  1. ...
  2. ...
- **预期**: [应该发生什么]
- **实际**: [实际发生什么]
- **根因猜测**: [可能的原因]
- **建议修复**: [修复方向]
```

---

## Step 4: 回归验证

Bug 修复后，重新执行完整测试场景：
1. 原始 Bug 是否修复？
2. 修复是否引入新问题？
3. 之前通过的场景是否仍然通过？
